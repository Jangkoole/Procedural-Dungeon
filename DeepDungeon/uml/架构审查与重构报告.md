## 2. 系统概述

### 2.1 项目简介

DeepDungeon 是一个纯控制台版本的 Roguelike 地牢游戏，采用 C++17 开发，核心功能包括：

- 程序化地图生成
- 战争迷雾 (FOV) 系统
- 回合制战斗
- 物品拾取与使用
- 怪物 AI 追踪

| 构建系统 | CMake 3.16+ |

## 3. 架构现状分析

### 3.1 静态类图分析

```
┌─────────────────────────────────────────────────────────┐
│                      Game (上帝类)                       │
│  - Map map_                                             │
│  - unique_ptr<Player> player_                           │
│  - vector<unique_ptr<Entity>> entities_                 │
│  - bool gameOver_, gameWon_                             │
│  - vector<string> log_                                  │
│                                                         │
│  + 10 个公有方法                                         │
│  - 7 个私有方法 (战斗、AI、FOV、碰撞、拾取...)            │
└─────────────────────────────────────────────────────────┘
```

**问题：** `Game` 类承担了过多职责，违反 **单一职责原则 (SRP)**。

### 3.2 继承层次分析

```
Entity (基类)
├── Player    ← 有独立行为 (addExp, usePotion)
├── Monster   ← 几乎无行为 (数据容器)
└── Item      ← 几乎无行为 (数据容器)
```

**问题：** 子类行为外泄到 `Game` 类，形成 **贫血模型**。

### 3.3 依赖关系分析

```
Game ──► Map      (组合，强依赖)
Game ──► Player   (组合，强依赖)
Game ──► Entity   (组合，强依赖)
Game ──► Monster  (直接使用，隐式依赖)
Game ──► Item     (直接使用，隐式依赖)
```

**问题：** `Game` 与所有具体类耦合，难以扩展新实体类型。

## 4. 代码坏味道识别

### 4.1 🔴 上帝类 (God Class)

**位置：** `Game.h` / `Game.cpp`

**症状：**
- 17 个成员变量/方法
- 7 个私有辅助方法
- 直接操作所有实体内部状态

**代码证据：**
```cpp
// Game.cpp - 攻击判定逻辑
void Game::attackMonster(Monster* monster) {
    int damage = player_->totalAtk();  // 读取玩家状态
    monster->takeDamage(damage);       // 修改怪物状态
    
    if (!monster->isAlive()) {
        player_->addExp(20);           // 修改玩家经验
        
        // 直接操作 entities_ 容器移除怪物
        entities_.erase(
            std::remove_if(entities_.begin(), entities_.end(),
                [monster](const std::unique_ptr<Entity>& e) {
                    return e.get() == monster;
                }),
            entities_.end()
        );
    }
}
```

**影响：**
- 难以单元测试
- 新增功能需修改核心类
- 代码可读性差

### 4.2 🔴 贫血模型 (Anemic Model)

**位置：** `Monster`, `Item` 类

**症状：**
- 仅有数据字段和 getter/setter
- 核心行为在 `Game` 中实现

**对比：**
```cpp
// 当前设计 - 贫血
class Monster : public Entity {
    Monster(int x, int y);  // 仅构造函数
};

// 理想设计 - 充血
class Monster : public Entity {
    void attack(Entity* target);
    void chase(Entity* target, Map* map);
    int calculateDamageTo(Entity* target);
};
```

### 4.3 🟡 深层耦合 (Deep Coupling)

**位置：** `Game::updateMonsters()`

**症状：**
- `Game` 需要了解 `Monster` 的内部移动逻辑
- 使用 `static_cast` 进行类型转换

**代码证据：**
```cpp
void Game::updateMonsters() {
    for (auto& entity : entities_) {
        if (entity->type() != EntityType::Monster) continue;
        
        Monster* monster = static_cast<Monster*>(entity.get());  // ⚠️ 类型转换
        // ... 直接操作 monster 位置
    }
}
```

### 4.4 🟡 违反开闭原则 (OCP)

**位置：** `Game::pickupItem()`

**症状：**
- 新增物品类型需修改 `switch` 语句

**代码证据：**
```cpp
void Game::pickupItem() {
    Item* item = getItemAt(player_->x(), player_->y());
    if (item) {
        switch (item->itemType()) {  // ⚠️ 需修改现有代码
            case ItemType::Weapon:
                player_->equipWeapon(5);
                break;
            case ItemType::Potion:
                player_->addPotion();
                break;
        }
    }
}
```

### 4.5 🟡 静态依赖

**位置：** `Map.cpp`

**症状：**
- 固定随机数种子 (`std::mt19937 rng(42)`)
- 难以进行可重复测试

### 4.6 🟢 魔法数字

**位置：** 多处

| 位置 | 魔法数字 | 建议 |
|------|----------|------|
| `Player::addExp()` | `50` (升级阈值) | 提取为常量 |
| `Game::updateMonsters()` | `5` (怪物数量) | 配置化 |
| `Map::generateRooms()` | `6` (房间数量) | 配置化 |

## 5. 重构方案设计

### 5.1 重构目标

| 目标 | 衡量指标 |
|------|----------|
| 消除上帝类 | `Game` 类方法数 < 5 |
| 充血模型 | `Monster`/`Item` 有独立行为方法 |
| 降低耦合 | 移除 `static_cast` |
| 符合 OCP | 新增实体无需修改 `Game` |

### 5.2 核心策略

#### 策略 1：引入命令模式 (Command Pattern)

**目的：** 解耦输入处理与游戏逻辑

```cpp
// 新增：Command 接口
class ICommand {
public:
    virtual ~ICommand() = default;
    virtual void execute(GameState& state) = 0;
};

// 具体命令
class MoveCommand : public ICommand {
    void execute(GameState& state) override {
        state.player.move(dx, dy);
        state.map.updateFOV(state.player.position());
    }
};

class AttackCommand : public ICommand {
    void execute(GameState& state) override {
        auto target = state.getEntityAt(targetPos);
        state.player.attack(target);  // 玩家自己执行攻击
    }
};
```

**收益：**
- `Game` 不再直接处理输入
- 新增命令无需修改 `Game`

#### 策略 2：实体行为内聚

**目的：** 消除贫血模型

```cpp
// 重构后：Monster 类
class Monster : public Entity {
public:
    void updateAI(GameState& state) {
        auto player = state.getPlayer();
        if (canSee(player)) {
            chase(player, state.map);
        }
    }
    
    void chase(Entity* target, Map* map) {
        // 寻路逻辑内聚在 Monster 中
    }
    
    int calculateDamageTo(Entity* target) override {
        return std::max(1, atk_ - target->def());
    }
    
    void attack(Entity* target) {
        int damage = calculateDamageTo(target);
        target->takeDamage(damage);
    }
};
```

#### 策略 3：引入事件系统

**目的：** 解耦战斗判定与游戏状态更新

```cpp
// 新增：事件系统
class EventSystem {
public:
    using Listener = std::function<void(const Event&)>;
    
    void subscribe(EventType type, Listener listener);
    void publish(Event event);
};

// 使用示例
class CombatSystem {
    void onAttack(AttackEvent& event) {
        int damage = attacker.calculateDamage(target);
        target.takeDamage(damage);
        
        if (!target.isAlive()) {
            eventSystem.publish(DeathEvent(target, killer));
        }
    }
};
```

#### 策略 4：引入状态模式管理游戏流程

```cpp
// 新增：游戏状态接口
class IGameState {
public:
    virtual void handleInput(Game& game, Command cmd) = 0;
    virtual void update(Game& game) = 0;
};

class PlayingState : public IGameState {
    void handleInput(Game& game, Command cmd) override {
        // 正常游戏逻辑
    }
};

class GameOverState : public IGameState {
    void handleInput(Game& game, Command cmd) override {
        // 只处理重新开始/退出
    }
};
```

### 5.4 面向对象原则应用

| 原则 | 应用场景 | 实现方式 |
|------|----------|----------|
| **封装** | 实体状态管理 | 将 HP/位置等封装在 Entity 内部 |
| **继承** | 实体类型扩展 | Entity → Player/Monster/Item |
| **多态** | 战斗行为 | 虚函数 `attack()`, `takeDamage()` |
| **依赖倒置** | 系统间通信 | 通过 `EventSystem` 解耦 |
