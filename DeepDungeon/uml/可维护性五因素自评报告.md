# 可维护性五因素自评报告

> **项目名称：** DeepDungeon（深度地牢）
> **评审日期：** 2026年4月23日
> **评审范围：** 全部源代码（`include/` + `src/`）
> **参考标准：** 教材 §8.4.1 可维护性五因素
> **评分制：** 1（极差）~ 5（优秀）

---

## 总体评分一览

| 因素 | 评分 | 严重程度 |
|------|:----:|:--------:|
| 可理解性（Understandability） | ⭐⭐⭐ **3** | 中等 |
| 可测试性（Testability） | ⭐ **1** | 🔴 严重 |
| 可修改性（Modifiability） | ⭐⭐ **2** | 🟡 较高 |
| 可移植性（Portability） | ⭐⭐⭐⭐ **4** | 🟢 良好 |
| 可重用性（Reusability） | ⭐⭐ **2** | 🟡 较高 |

---

## 一、可理解性（Understandability）—— 评分：3/5

### 评分依据

**得分项（+）：**
- 命名规范良好：类名（`Game`、`Map`、`Player`、`Entity`）、方法名（`takeDamage`、`isWalkable`、`updateFOV`）语义清晰
- 代码结构层次分明：头文件与实现分离，`include/` 和 `src/` 目录划分合理
- 常量集中管理：`Constants.h` 统一存放游戏常量，便于查阅
- 注释虽少但关键处有说明（如 `// 简单追踪 AI`、`// 简单圆形 FOV`）

**失分项（-）：**
- `Game` 类职责过重（17 个方法），阅读时难以快速定位某个功能的完整逻辑链
- `Entity::takeDamage()` 中防御计算逻辑与调用方 `Game::attackMonster()` 中的伤害计算逻辑不一致：
  - `Entity::takeDamage()` 内部已做 `damage - def_` 计算
  - 但 `Game::attackMonster()` 直接传入 `player_->totalAtk()` 作为伤害值，未考虑怪物防御
  - 而 `Game::updateMonsters()` 中怪物攻击玩家时又手动计算 `max(1, monster->atk() - player_->def())`
  - **同一伤害计算逻辑分散在三处，行为不一致**
- `Map::generateCorridors()` 中走廊连接算法逻辑不够直观，缺少注释说明连接策略
- 魔法数字散布各处（`50` 升级阈值、`5` 怪物数量、`3` 物品数量、`20` 击杀经验值、`42`/`456`/`123` 随机种子）

### 改进方案

| 问题 | 改进措施 | 工作量 |
|------|----------|:------:|
| 伤害计算逻辑不一致 | 统一为 `CombatSystem` 模块，封装 `calculateDamage(attacker, defender)` 函数 | 小 |
| 魔法数字 | 提取为 `Constants.h` 中的命名常量（`EXP_PER_LEVEL`、`INITIAL_MONSTER_COUNT`、`INITIAL_ITEM_COUNT`、`MONSTER_KILL_EXP` 等） | 小 |
| 走廊算法不直观 | 补充算法注释，说明 L 形走廊的连接策略 | 极小 |
| Game 上帝类 | 拆分职责（见可修改性改进方案） | 大 |

## 二、可测试性（Testability）—— 评分：1/5

### 评分依据

**得分项（+）：**
- 部分函数是纯逻辑，理论上可测：`Entity::takeDamage()`、`Player::addExp()`、`Player::usePotion()`、`Map::isWalkable()`

**失分项（-）：**
- **无任何单元测试框架**，项目中没有测试文件、没有测试目录、没有测试依赖
- **硬依赖不可替换**：`Game` 类直接持有 `Map`、`Player`、`Entity` 的具体类型，无法注入 mock/stub
- **随机数不可控**：`Map::generateRooms()` 使用固定种子 `42`，但 `Game::init()` 中使用 `456`，`Map::addWaterFeatures()` 中使用 `123`，测试时无法统一控制随机行为
- **`Game` 类职责过重**：输入处理、游戏逻辑、渲染、日志全部耦合在一起，无法单独测试战斗逻辑或怪物 AI
- **控制台 I/O 耦合**：`main.cpp` 直接使用 `std::cin`/`std::cout`，无法自动化测试游戏流程
- **`Map::generate()` 无确定性输出保证**：虽然用了固定种子，但生成结果依赖多个随机分布，测试断言困难
- **`test_run.bat` 仅做冒烟测试**：通过管道输入模拟按键，但无任何断言验证输出结果

### 改进方案

| 问题 | 改进措施 | 工作量 |
|------|----------|:------:|
| 无测试框架 | 引入 Google Test 或 Catch2 作为测试框架，配置 CMake 测试目标 | 中 |
| 硬依赖不可替换 | 为 `Map`、`Entity` 等核心类提取接口（抽象基类），`Game` 通过依赖注入持有接口指针 | 大 |
| 随机数不可控 | 将随机数引擎作为参数传入生成函数（`Map::generate(RandomEngine& rng)`），测试时传入固定种子引擎 | 中 |
| Game 职责过重 | 拆分后各子系统可独立测试（`CombatSystem`、`MovementSystem`、`ItemSystem`） | 大 |
| 控制台 I/O 耦合 | 引入 `InputHandler` 和 `OutputRenderer` 接口，测试时替换为 mock | 中 |
| 无断言验证 | 使用 Google Test 的 `EXPECT_EQ`/`EXPECT_TRUE` 等断言宏 | 小 |

**建议首批添加的单元测试（3 个）：**

1. **`EntityTest::TakeDamage`** — 验证 `takeDamage()` 的伤害减免计算和 `heal()` 的回血上限
2. **`PlayerTest::AddExp`** — 验证经验值累积和升级阈值（50 exp/级），以及升级后属性增长
3. **`PlayerTest::UsePotion`** — 验证药水数量减少和 HP 恢复逻辑

## 三、可修改性（Modifiability）—— 评分：2/5

### 评分依据

**得分项（+）：**
- 常量集中管理，修改地图尺寸、视野半径等只需改 `Constants.h`
- 类层次结构清晰，新增实体类型只需继承 `Entity`

**失分项（-）：**
- **违反开闭原则（OCP）**：`Game::pickupItem()` 中使用 `switch(item->itemType())` 分支处理物品类型，新增物品类型（如 armor、scroll）必须修改 `Game` 类
- **`Game` 上帝类**：修改任何一个功能（战斗、AI、物品、FOV）都需要修改 `Game`，修改风险高、影响范围大
- **`static_cast` 类型转换**：`Game::updateMonsters()` 和 `getMonsterAt()`/`getItemAt()` 中使用 `static_cast<Monster*>` 和 `static_cast<Item*>`，类型不安全，新增实体子类时容易遗漏转换
- **`setAtk` 自赋值 Bug**：`Entity::setAtk(int atk)` 中写的是 `atk_ = atk_;`（自赋值），导致无法修改攻击力
- **`UsePotion` 命令未实现**：`main.cpp` 的 `switch` 中 `case Command::UsePotion:` 只有 `break`，药水使用逻辑在 `Game::update()` 中并未被触发
- **怪物数量硬编码**：`Game::init()` 中 `for (int i = 0; i < 5; i++)`，修改怪物数量需改源码
- **物品效果硬编码**：武器固定 +5 ATK，药水固定 +25 HP，修改效果需改 `Game::pickupItem()` 和 `Player::usePotion()`

### 改进方案

| 问题 | 改进措施 | 工作量 |
|------|----------|:------:|
| pickupItem 违反 OCP | 引入策略模式：`ItemEffect` 接口，每种物品类型实现自己的 `apply(Player&)` 方法 | 中 |
| Game 上帝类 | 拆分为 `CombatSystem`、`MonsterAISystem`、`ItemSystem`、`FOVSystem`、`RenderSystem` | 大 |
| static_cast 类型不安全 | 使用多态虚函数替代：在 `Entity` 中定义 `virtual void onPlayerInteract(Player&)`，子类各自实现 | 中 |
| setAtk 自赋值 Bug | 修复为 `atk_ = atk;` | 极小 |
| UsePotion 未实现 | 在 `Game::update()` 中检查 `pendingPotionUse_` 标志并调用 `player_->usePotion()` | 小 |
| 硬编码数值 | 提取为 `Constants.h` 中的命名常量 | 小 |

## 四、可移植性（Portability）—— 评分：4/5

### 评分依据

**得分项（+）：**
- **纯 C++17 标准**：无平台特定 API 依赖（无 Win32 API、无 POSIX 调用）
- **CMake 构建系统**：跨平台构建工具，支持 Windows/Linux/macOS
- **控制台 I/O**：仅使用 `std::cin`/`std::cout`，标准 C++ 流
- **已预留 curses 选项**：`CMakeLists.txt` 中有 `USE_CURSES` 选项，未来可启用 ncurses/PDCurses 增强终端控制
- **无外部依赖**：纯标准库实现，无需安装第三方库

**失分项（-）：**
- **Windows 编码硬编码**：`main.cpp` 中 `system("chcp 65001 > nul")` 是 Windows 特有命令，Linux/macOS 下会报错
- **`#ifdef _WIN32` 预处理**：虽然做了条件编译，但 Linux 下 UTF-8 通常已是默认编码，无需额外设置
- **路径分隔符**：`test_run.bat` 是 Windows 批处理脚本，Linux/macOS 下无法直接使用
- **MinGW 路径硬编码**：`.vscode/tasks.json` 中 `"E:\\C++\\mingw64\\bin\\g++.exe"` 是特定机器路径

### 改进方案

| 问题 | 改进措施 | 工作量 |
|------|----------|:------:|
| Windows 编码硬编码 | 保留 `#ifdef _WIN32` 条件编译，Linux 下可添加 `setlocale(LC_ALL, "")` 作为替代 | 极小 |
| test_run.bat 不可移植 | 同时提供 `test_run.sh` shell 脚本，或在 CMake 中添加 `add_test` 命令 | 小 |
| tasks.json 路径硬编码 | 使用 CMake 预设（CMakePresets.json）替代 VS Code 任务，或使用 `${workspaceFolder}` 相对路径 | 小 |

## 五、可重用性（Reusability）—— 评分：2/5

### 评分依据

**得分项（+）：**
- `Entity` 基类设计合理，位置、属性、移动等基础功能可独立复用
- `Map` 类的地图生成算法（房间+走廊+水域）可独立提取为地图生成库
- `Constants.h` 中的常量定义可被任何需要地牢配置的项目引用

**失分项（-）：**
- **`Game` 上帝类**：游戏逻辑与渲染、输入、日志深度耦合，无法单独复用战斗系统或 AI 系统
- **贫血模型**：`Monster` 和 `Item` 仅有构造函数，行为全部在 `Game` 中，无法独立复用
- **`Map` 与 `Game` 耦合**：`Map` 声明了 `friend class Game`，破坏了封装性，`Game` 可直接访问 `Map` 的私有成员
- **日志系统内嵌**：日志逻辑直接写在 `Game` 中，无法独立复用为日志模块
- **渲染逻辑内嵌**：`Game::render()` 直接拼接字符串，无法替换为不同的渲染后端（如 curses、GUI）

### 改进方案

| 问题 | 改进措施 | 工作量 |
|------|----------|:------:|
| Game 上帝类 | 拆分为独立子系统，每个子系统可单独链接到其他项目 | 大 |
| 贫血模型 | 将行为移入实体类（`Monster::act(Player&, Map&)`、`Item::applyTo(Player&)`） | 中 |
| friend class Game | 为 `Map` 提供公有接口替代友元访问，移除 `friend` 声明 | 中 |
| 日志系统 | 提取为独立的 `Logger` 类，支持多级别日志和输出目标替换 | 小 |
| 渲染系统 | 提取 `Renderer` 接口，支持控制台字符串渲染和 curses 渲染两种实现 | 中 |
